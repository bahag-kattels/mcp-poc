Slide 1

Why this exists

MCP servers should be deployed as narrowly scoped, well-observed services behind an API gateway.

The goal is not to expose a general-purpose backend. The goal is to expose a controlled tool interface that agents can use safely.

Slide 2

Recommended deployment model

  • Run the MCP server in a separate Cloud Run service or Cloud Run function.
  • Keep it isolated from the product application backend.
  • Use a minimal server implementation such as ../mcp-python/mcp-barebones.py.

Slide 3

Apigee routing pattern

Route only the MCP path to the MCP target. Keep the default proxy path on the normal backend.

<RouteRule name="to-mcp">
  <Condition>(proxy.pathsuffix MatchesPath "/mcp/**")</Condition>
  <TargetEndpoint>mcp-target</TargetEndpoint>
</RouteRule>

<RouteRule name="default">
  <TargetEndpoint>default-target</TargetEndpoint>
</RouteRule>

  region     = "europe-west1"

  paths = [
    {
      path         = "/v2/product-recommendation/5/mcp/*"
      backend_path = "/"
    }
  ]
}

Slide 5

Caveats for agents

  • Treat MCP tools as constrained actions, not open-ended application access.
  • Assume every tool call may be repeated, retried, or malformed.
  • Never rely on the agent to infer business policy from the model prompt alone.
  • Do not expose secrets, credentials, or privileged internal endpoints through MCP.
  • Keep tool outputs small, explicit, and deterministic where possible.
  • Fail closed when validation, auth, or routing is ambiguous.

Slide 6

Required guardrails

  • Path-level isolation so only /mcp/* reaches the MCP backend.
  • Input validation for all tool arguments and payloads.
  • Strict timeout and size limits for requests and responses.
  • Rate limiting and abuse protection at the gateway.
  • Structured logging and traceability for every tool invocation.
  • Safe defaults for retries, error handling, and partial failures.

Slide 7

What product teams should keep in mind

  • Define the exact job the agent is allowed to do before exposing a tool.
  • Decide what the tool must never do, and encode that in the gateway and backend.
  • Design for narrow workflows, not broad autonomy.
  • Document ownership: who maintains the tools, schemas, and routing rules.
  • Plan versioning and compatibility before changing any tool contract.
  • Make failure modes visible to users and support teams.
  • Treat observability as product infrastructure, not optional logging.
